## 1. Summary

在线达标助手是一款面向电商在线客服的 Windows 桌面计时工具。它记录用户每天的有效在线时长和挂起时长，根据早班、中班、晚班计算在线完成率、挂起余量、补时和预计完成时间，并用主界面、悬浮窗和系统托盘持续反馈。

产品首先解决“下班前不知道是否达标”的问题。V1 以本地离线使用为主，不读取客服平台内容，不替代企业考勤系统。

## 3. Background & Context

### 3.1 问题

客服每天需要累计在线 7 小时 30 分钟。当前班次为：

| 班次 | 时间 | 班次总长 | 按当前口径可用挂起参考 |
| --- | --- | ---: | ---: |
| 早班 | 08:30-17:00 | 8 小时 30 分 | 1 小时 |
| 中班 | 11:00-20:00 | 9 小时 | 1 小时 30 分 |
| 晚班 | 14:00-22:00 | 8 小时 | 30 分钟 |

班次中除去 7.5 小时在线的剩余时间可以自行安排吃饭和休息。用户需要在工作中频繁切换在线与挂起，靠记忆或手工计算容易出现以下问题：

- 不知道今日在线是否已经达标。
- 不知道挂起是否超过可用参考。
- 未达标时不知道还要补多久。
- 应用关闭、电脑重启或跨天后容易丢记录。
- 无法按周、按月复盘自己的时间情况。

### 3.2 为什么现在做

这是一个高频、低复杂度但强依赖准确性的个人工具场景。用户已经有明确的班次和在线目标，产品可以先通过本地计时解决核心问题，不需要等待客服平台接口或企业系统改造。

### 3.3 约束和口径

1. “挂起参考”用于提醒，不自动限制用户，也不直接代表公司正式考勤结论。
2. 只有在线状态计入有效在线；挂起状态不计入在线。
3. 默认使用系统本地时间，核心功能离线可用。
4. 历史记录保存当日班次快照，未来配置变化不能悄悄改写历史结果。

## 4. Objectives & Success Metrics

### 4.1 Goals

1. 让用户在 3 秒内看到当前状态、有效在线、挂起、补时和预计完成时间。
2. 让用户用一次点击完成在线/挂起切换，减少手工记录。
3. 让应用重启、托盘运行和跨天时保持数据连续、可解释。
4. 让用户可以查看周、月和历史数据，并导出可计算的 Excel 文件。

### 4.2 Non-goals

1. 不自动读取、控制或模拟第三方客服平台状态。
2. 不做团队账号、云同步、权限管理和组长后台。
3. 不替代企业考勤、薪资或请假审批系统。
4. 不采集客户聊天内容、屏幕内容、键盘输入或客服业务数据。

### 4.3 SMART Success Metrics

| 指标 | 基线 | 目标 | 测量方式 |
| --- | --- | --- | --- |
| 首次配置完成率 | 待测 | 90% 用户在 1 分钟内完成班次确认并开始计时 | 首次使用测试 |
| 核心流程完成率 | 待测 | 95% 用户可完成在线、挂起、恢复、查看结果、导出 | 可用性测试 |
| 计时准确性 | 无工具基线 | 正常运行时状态切换误差不超过 1 秒 | 自动化时间测试 |
| 日记录保存成功率 | 无工具基线 | 99.9% 的状态切换成功持久化 | 本地诊断日志 |
| 异常恢复成功率 | 无工具基线 | 95% 的重启、锁屏和跨天场景可恢复或明确请求确认 | 恢复测试 |
| 周活跃使用率 | 待测 | 发布后 30 天内，至少 60% 的安装用户每周使用 3 天 | 本地匿名统计或用户调研 |

## 5. Target Users & Segments

产品按用户要完成的工作来定义市场，而不是按年龄或行业泛化。

### 5.1 核心用户：需要累计在线时长的一线客服

**要完成的工作**：在班次内完成规定在线时间，同时安排吃饭、休息和短暂离开。  
**主要痛点**：频繁切换状态，无法记住每次挂起；下班前需要人工计算。  
**成功标准**：无需计算就知道“已在线多久、还要多久、挂起是否超出”。

### 5.2 次要用户：需要个人复盘的客服

**要完成的工作**：查看一周或一个月的达标情况，发现经常补时的日期。  
**主要痛点**：历史数据零散，无法比较不同班次。  
**成功标准**：能够按日期和班次查看，并导出准确数据。

### 5.3 约束

- 用户通常同时使用客服系统，主工具不能遮挡工作区域。
- 用户不应被复杂设置打断，默认班次必须开箱可用。
- 用户可能锁屏、睡眠或重启电脑，计时恢复必须可解释。
- 颜色、数字和文字要同时表达状态，避免只靠颜色判断。

## 6. Value Propositions

### 6.1 用户任务与收益

| 用户任务 | 产品提供的价值 |
| --- | --- |
| 开始工作 | 自动提示星期和班次，减少配置判断 |
| 吃饭或休息 | 一键进入挂起，自动累计挂起时长 |
| 回到岗位 | 一键恢复在线，继续累计有效在线 |
| 下班前判断 | 圆环、补时倒计时和预计完成时间直接给答案 |
| 每周复盘 | 周/月/历史统计展示达标和补时趋势 |
| 汇报或留档 | Excel 导出保留可计算的明细 |

### 6.2 需要避免的损失

- 因忘记恢复在线而误以为已经达标。
- 因挂起过多导致临近下班才发现需要补时。
- 因应用重启、关闭主窗或跨天导致记录丢失。
- 因历史统计口径不一致而无法复盘。

### 6.3 产品价值曲线

相对纸笔、备忘录和普通倒计时工具，本产品重点提升：

- **口径清楚**：把班次、有效在线、挂起和补时分开。
- **持续可见**：托盘和悬浮窗让用户不用反复打开主界面。
- **可恢复**：本地保存状态区间，而不是只保存一个最终数字。
- **可复盘**：按日、周、月生成同一口径的历史数据。

产品不追求自动监控用户行为，也不把提醒设计成强制考勤。

## 7. Solution Overview

### 7.1 UX / User Flow

```text
启动应用
  -> 识别本地日期和星期
  -> 匹配今日班次或显示休息日
  -> 用户确认班次
  -> 开始在线
  -> 吃饭/休息时开始挂起
  -> 回岗时恢复在线
  -> 实时查看在线圆环、挂起余量和补时
  -> 达标或结束今日记录
  -> 自动保存并进入统计
```

#### 主界面信息布局

```text
┌──────────────────────────────────────────────┐
│ 在线达标助手   2026-07-14 周二   早班  设置 主题 │
├──────────────────────┬───────────────────────┤
│                      │ 当前状态：在线         │
│      进度圆环         │ 今日在线：05:23/07:30 │
│        71%           │ 已用挂起：00:42       │
│                      │ 挂起余量：00:18       │
│                      │ 预计完成：16:48       │
├──────────────────────┴───────────────────────┤
│ 时间轴：在线 ━━ 挂起 ━ 在线 ━━━━━━━━━          │
│              [开始挂起]                         │
└──────────────────────────────────────────────┘
```

#### 关键交互

- 未开始：显示“开始在线”。
- 在线：显示“开始挂起”。
- 挂起：显示“恢复在线”。
- 已达标：显示“今日在线已达标”，仍允许继续记录超额在线。
- 关闭主窗口：默认进入托盘，不退出计时。
- 悬浮窗：显示状态、在线时长、补时和挂起时长，支持快速切换。

### 7.3 计时与数据规则

#### 计算公式

- 目标在线时长：默认 `07:30:00`。
- 有效在线时长：在线区间与有效班次区间的交集之和。
- 可用挂起参考：`max(0, 班次时长 - 目标在线时长)`。
- 在线完成率：`min(有效在线 / 目标在线, 100%)`。
- 补时剩余：`max(0, 目标在线 - 有效在线)`。
- 挂起余量：`可用挂起参考 - 已用挂起`。

#### 默认口径

1. 只有在线状态计入有效在线。
2. 班次前启动在线时，默认从班次开始计入。
3. 普通班次结束后不再自动增加当日有效在线；是否允许班次外补时由设置决定。
4. 跨午夜班次归属到开始日期。
5. 达标后圆环保持 100%，另外显示超额在线。

### 7.4 Technology

技术方案可调整，但必须满足以下要求：

- 使用 Windows 桌面容器，优先考虑 Tauri 2 或同等轻量方案。
- 前端使用 TypeScript 和稳定的组件系统，保持 Windows 11 风格。
- 使用本地 SQLite 保存设置、每日记录和状态区间。
- Excel 在本地生成，不依赖网络。
- 支持系统托盘、通知、窗口置顶、开机启动和单实例锁。
- 空闲时 CPU 目标小于 1%，内存目标小于 150 MB。

### 7.5 Assumptions

以下假设需要在真实用户测试中验证：

1. 用户愿意在开始工作和休息时点击一次按钮。
2. 用户的主要需求是个人记录，而不是企业强制考勤。
3. “挂起参考 = 班次时长 - 7.5 小时”符合当前业务的提醒口径。
4. 本地存储足以满足隐私和使用需求，暂不需要账号与云同步。
5. 锁屏、睡眠和关机后的计时恢复可以通过用户确认解决。

## 8. Timeline & Phasing

采用相对时间，不绑定具体日历日期。

### Phase 0：口径确认与原型（约 3-5 个工作日）

- 确认挂起定义、班次外补时、睡眠恢复和周起始日。
- 完成主界面、统计页、设置页和悬浮窗低保真原型。
- 用 2-3 名真实客服验证按钮位置和信息优先级。

### Phase 1：V1 核心版本（约 2-3 周）

- 完成 P0 全部功能：计时、班次、圆环、补时、挂起参考、预计完成、自动保存、跨天、统计、Excel、托盘和悬浮窗。
- 完成本地数据结构、迁移、异常恢复和单实例处理。
- 完成浅色/深色主题和核心可访问性。

### Phase 2：V1.1 体验完善（约 1-2 周）

- 完成 P1：自定义班次、提醒细化、手工修正、数据备份、统计筛选优化。
- 进行 Windows 11 不同缩放比例和多显示器测试。
- 完成安装包、卸载、升级和用户帮助文案。

### Phase 3：发布验证（约 3-5 个工作日）

- 小范围灰度给真实客服使用。
- 重点观察计时恢复、跨天、误操作修正和 Excel 数据一致性。
- 达成 P0 验收标准后发布 V1；未确认的业务口径不得默默写死。

### 后续版本

- V2 再评估团队账号、云同步、客服平台状态同步和权限管理。
- 只有在 V1 证明个人计时工具有稳定使用需求后，才扩展团队能力。
