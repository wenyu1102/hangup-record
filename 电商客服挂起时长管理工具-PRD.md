# 电商客服在线时长与挂起管理工具 PRD

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 挂起记录 |
| 文档版本 | V1.0 |
| 文档状态 | 生产需求文档 |
| 目标平台 | Windows 11 桌面端 |
| 目标用户 | 电商在线客服、客服组长（个人使用优先） |
| 核心目标 | 记录每日在线与挂起时长，实时判断是否需要补时，降低人工计算成本 |

## 2. 产品背景与问题

电商客服每天需要在班次时间范围内保持累计 7.5 小时在线。班次之外的时间可自主用于吃饭、休息等安排，但目前缺少可靠的计时与统计工具，用户下班前无法快速判断：

1. 今日累计在线时长是否已达到 7.5 小时。
2. 今日挂起时长是否超出可用余量。
3. 如果尚未达标，需要补充多久，以及预计何时可以下班。
4. 本周、本月和历史的在线达标情况。

产品通过桌面计时、悬浮窗和托盘入口自动记录状态，并将每日数据自动保存，提供清晰的实时反馈和可导出的统计结果。

## 3. 产品目标与成功标准

### 3.1 产品目标

- 用户在 3 秒内看到当前状态、今日在线时长、挂起时长和是否需要补时。
- 用户无需手工记录每次挂起，计时结果可在应用重启后恢复。
- 系统根据星期几自动匹配班次，并允许用户覆盖为自定义班次。
- 用户可查看周、月、历史统计，并导出 Excel。
- 应用在关闭主窗口后仍可通过系统托盘继续运行。

### 3.2 成功标准

- 计时误差：正常运行状态下单次状态切换误差不超过 1 秒；应用重启后按时间戳恢复。
- 数据可靠性：发生状态变化、应用退出和定时周期保存时均持久化数据；异常退出后最多丢失一个保存周期内的数据。
- 首次使用：用户在 1 分钟内完成班次确认并开始计时。
- 核心任务完成率：用户能够独立完成“开始在线 -> 挂起 -> 恢复 -> 查看是否补时 -> 导出”完整流程。

## 5. 核心概念与计时口径

### 5.1 核心概念

| 概念 | 定义 |
| --- | --- |
| 班次时段 | 当天允许工作的时间范围，例如 08:30-17:00 |
| 目标在线时长 | 用户每天必须累计在线的时长，默认 7 小时 30 分钟 |
| 在线 | 用户正在计入有效在线时长的状态 |
| 挂起 | 用户暂时不计入在线时长的状态，例如吃饭、休息、离开电脑 |
| 已用挂起 | 当日所有挂起区间的累计时长 |
| 可用挂起 | 班次时长减目标在线时长；只作为提醒参考，不限制用户挂起 |
| 补时 | 截止当前仍需累计的在线时长，公式为目标在线时长减有效在线时长 |
| 预计下班时间 | 按当前时间和未完成在线目标推算的预计完成时间 |

### 5.2 默认班次

| 班次 | 时间 | 班次长度 | 默认可用挂起参考 |
| --- | --- | ---: | ---: |
| 早班 | 08:30-17:00 | 8 小时 30 分 | 1 小时 |
| 中班 | 11:00-20:00 | 9 小时 | 1 小时 30 分 |
| 晚班 | 14:00-22:00 | 8 小时 | 30 分钟 |

默认目标在线时长统一为 7 小时 30 分。可用挂起参考按“班次长度 - 目标在线时长”计算，不等同于公司硬性考勤规则。若用户自定义目标在线时长，系统重新计算该参考值。

### 5.3 计时原则

1. 只有在用户点击“开始在线”或“恢复在线”后，时间才计入有效在线时长。
2. 点击“开始挂起”后，时间停止计入在线时长，并计入挂起时长。
3. 未开始计时、暂停记录、应用未运行的时间不自动推断为在线或挂起。
4. 计时只在有效班次范围内计入；班次前启动在线时，默认从班次开始时间计入；班次结束后如果在线不足，继续累计在线直到目标完成。
5. 跨午夜班次采用业务日期归属规则：跨天班次归属到开始日期，具体结束时间按次日计算。
6. 状态切换按时间戳记录，不依赖界面刷新频率。

### 5.4 关键计算公式

设：

- `目标在线时长` = 7.5 小时。
- `有效在线时长` = 所有有效在线区间与班次有效区间的交集之和。
- `已用挂起时长` = 所有挂起区间时长之和。
- `班次时长` = 班次结束时间 - 班次开始时间。
- `可用挂起时长` = max(0, 班次时长 - 目标在线时长)。

则：

- 在线完成率 = min(有效在线时长 / 目标在线时长, 100%)。
- 补时剩余 = max(0, 目标在线时长 - 有效在线时长)。
- 挂起余量 = 可用挂起时长 - 已用挂起时长。
- 当前是否超出挂起参考 = 已用挂起时长 > 可用挂起时长。
- 预计完成时间 = 当前时间 + 补时剩余；若当前为挂起状态，默认显示“恢复在线后预计完成”，不把挂起时间算作有效在线。

当有效在线时长达到目标后，补时剩余显示 `已达标`，预计下班时间显示达标时间；之后继续在线仍可记录，但在线完成率保持 100%，并额外显示“超额在线”。

## 6. 用户角色与典型场景

### 6.1 个人客服

- 上班后确认系统识别的班次，点击开始在线。
- 吃饭或休息时点击挂起，回来后点击恢复在线。
- 下班前查看圆环、补时倒计时和挂起提醒。
- 在统计页查看本周、本月表现并导出 Excel。

### 6.2 客服组长（非账号角色）

V1.0 不提供团队权限。组长可接收用户主动导出的 Excel 文件，用于个人复盘或人工汇总。

## 7. 信息架构与页面

### 7.1 主界面

主界面是默认打开页面，适合高频查看和操作。

布局建议：

- 顶部：产品名称、当前日期、星期、班次标签、设置入口、主题切换。
- 左侧主区域：工作进度圆环、在线状态、今日在线时长、目标时长。
- 右侧信息区：已用挂起、可用挂起、挂起余量、预计下班、补时倒计时。
- 底部操作区：开始在线/恢复在线、开始挂起、结束今日记录。
- 底部辅助区：今日时间轴、最近一次状态变化、数据保存状态。

### 7.2 统计页面

- Tab：本周、本月、历史。
- 汇总指标：目标在线总时长、实际在线总时长、达标天数、未达标天数、累计补时、累计挂起。
- 图表：每日在线时长柱状图、在线达标率折线图、挂起时长趋势图。
- 表格：日期、星期、班次、目标在线、实际在线、挂起、补时、状态、备注。
- 操作：按日期范围筛选、按班次筛选、导出 Excel。

### 7.3 设置页面

分为以下模块：

1. 班次设置：添加、编辑、删除、启用/停用班次；为星期几绑定班次。
2. 目标设置：默认每日目标在线时长，精确到分钟；支持按星期覆盖。
3. 提醒设置：班次开始提醒、挂起余量不足提醒、超出挂起参考提醒、达标提醒、补时提醒。
4. 显示设置：深色/浅色/跟随系统；是否显示悬浮窗；悬浮窗透明度和位置。
5. 托盘设置：关闭主窗口时最小化到托盘；开机自启动；托盘气泡提醒。
6. 数据设置：自动保存周期、数据文件位置、导出格式、备份和恢复。
7. 关于：版本号、检查更新、用户反馈入口。

### 7.4 悬浮窗

悬浮窗是可选的小型置顶窗口，默认显示：

- 当前状态：在线/挂起/未开始。
- 今日在线：`05:23 / 07:30`。
- 补时：`还需 02:07` 或 `已达标`。
- 挂起：`已用 00:42`。
- 快捷操作：切换在线/挂起、展开主界面、关闭悬浮窗。

悬浮窗支持拖动、置顶、记忆位置；不得遮挡其他软件的主要操作区域。用户可在设置中关闭。

## 8. 功能需求

### FR-01 自动识别星期和班次

**需求描述**：应用启动或日期变化时读取系统本地日期和时间，显示星期几并匹配当日班次。

**规则**：

- 默认星期一至星期日均可绑定早班、中班、晚班或休息日。
- 当天无班次时显示“今日休息”，不启动补时倒计时。
- 若当天配置多个班次，允许用户在主界面选择实际班次。
- 用户可关闭“自动提示班次”，关闭后不弹出提示，但主界面仍显示当前班次。
- 班次变更只影响新开始的计时；已经记录的区间保留原班次快照。

**验收标准**：修改系统日期或模拟跨天后，应用能显示正确星期和班次；无班次日不产生错误补时。

### FR-02 开始、挂起、恢复和结束记录

**状态机**：

`未开始 -> 在线 -> 挂起 -> 在线 -> 已结束`

允许从“未开始”直接结束，允许从“挂起”结束。已结束后当天默认只读，如需修改应通过历史记录编辑。

**按钮行为**：

- 未开始：主按钮显示“开始在线”。
- 在线：主按钮显示“开始挂起”。
- 挂起：主按钮显示“恢复在线”。
- 达标后：显示“已达标”，仍允许继续记录在线或挂起。
- 结束今日记录：弹出确认，并展示当前在线、挂起和补时数据。

结束确认文案应明确指出：结束后当天不会继续累计，是否结束。

### FR-03 工作进度圆环

- 圆环中心显示完成百分比，保留整数百分比。
- 圆环下显示 `实际在线 / 目标在线`，格式为 `05:23 / 07:30`。
- 未开始显示 0%；在线时实时更新；挂起时暂停在线增加但界面继续刷新。
- 达标后圆环显示 100%，并增加“已达标”状态；超额时显示超额在线时长。
- 状态颜色：未开始为中性灰，在线为蓝色，挂起为橙色，已达标为绿色，超出挂起参考为红色提示。

### FR-04 挂起时长管理

- 显示班次总长、目标在线、可用挂起参考、已用挂起和剩余挂起参考。
- 挂起倒计时从开始挂起时立即启动。
- 已用挂起低于参考值时显示“余量 X”；达到参考值时显示“已用完”；超出时显示“已超出 X”。
- 超出挂起参考只做提醒，不自动停止挂起，也不强制结束工作。
- 支持在历史记录中手工新增、编辑、删除挂起区间，并要求记录变更原因。

### FR-05 今日补时倒计时

- 补时剩余 = 目标在线 - 有效在线。
- 在线状态下每秒更新；挂起状态下补时数字保持不变并显示“当前挂起中”。
- 补时剩余大于 0 时使用倒计时格式 `还需 02:07:35`。
- 达标后显示“今日在线已达标”。
- 班次结束仍未达标时显示“班次已结束，仍需 X”，并允许用户继续记录补时。

### FR-06 预计下班时间

- 在未达标时显示预计完成时刻，例如“预计 17:42 完成”。
- 计算以当前时间为起点，在线状态按 1:1 增加；挂起状态不推进完成时间。
- 若当前时间早于班次开始，显示“班次开始后预计完成”。
- 若已达标，显示实际达标时间。
- 若没有开始记录，不显示虚假时间，显示“开始在线后计算”。
- 预计时间应同步考虑用户手工设置的固定休息时段（如配置了午休排除时段）。

### FR-07 自动保存与恢复

- 默认每 30 秒保存一次，并在状态切换、窗口关闭、系统关机通知时立即保存。
- 保存失败时在主界面显示非阻塞告警，并重试；不得静默丢失当前计时。
- 应用重新打开后，根据最后保存的状态和时间戳补齐可恢复的连续区间；若无法确认应用是否运行，按用户配置选择“继续计时”或“标记为未记录”。
- 默认策略：已开始的在线或挂起区间在应用重启后继续；锁屏、睡眠和关机状态不自动推断，用户恢复后主动选择继续在线或挂起。
- 数据文件保存在用户本地应用数据目录，并支持用户查看存储位置。

### FR-08 自动跨天重置

- 系统在本地日期变化时自动结束前一业务日未结束的当前区间，并生成日结快照。
- 新日期加载对应班次并将在线、挂起、补时清零。
- 跨天前若应用正在计时，应先保存前一天数据，再初始化新一天。
- 若用户跨午夜仍在工作，按跨天班次配置处理；普通非跨天班次不把次日时间计入前一天。

### FR-09 周、月和历史统计

- 周统计默认按自然周，周一至周日；允许在设置中切换周起始日。
- 月统计按自然月；历史支持自定义起止日期。
- 每日记录状态：未开始、进行中、已达标、未达标、休息日、手工修正。
- 统计数据应按每日保存的业务快照计算，不因后续班次配置变化而改变历史结果；用户编辑历史后重新生成快照。
- 对缺失数据显示“未记录”，不能默认当作 0 或达标。

### FR-10 Excel 导出

导出入口位于统计页，支持当前筛选结果和完整日期范围。

Excel 至少包含以下列：

| 字段 | 说明 |
| --- | --- |
| 日期 | 业务日期，格式 YYYY-MM-DD |
| 星期 | 中文星期 |
| 班次 | 早班/中班/晚班/自定义 |
| 班次开始 | HH:mm |
| 班次结束 | HH:mm |
| 目标在线时长 | 分钟或 HH:mm |
| 实际在线时长 | 分钟或 HH:mm |
| 可用挂起参考 | HH:mm |
| 实际挂起时长 | HH:mm |
| 补时/超额在线 | 未达标为补时，达标后为超额 |
| 达标状态 | 已达标/未达标/未记录/休息日 |
| 首次开始时间 | 当日第一次开始在线时间 |
| 实际达标时间 | 达标时记录 |
| 最后状态 | 导出时最后状态 |
| 手工修正 | 是/否 |
| 备注 | 用户备注 |

导出要求：

- 默认导出 `.xlsx`，文件名为 `在线时长统计_YYYYMMDD_HHmmss.xlsx`。
- 使用系统保存对话框选择路径。
- 导出成功显示文件路径，并提供打开文件夹入口。
- 无数据时禁止导出并提示先选择有记录的日期范围。
- 中文表头、冻结首行、自动列宽、时间列使用可计算的数值格式。

### FR-11 深色/浅色主题

- 支持浅色、深色、跟随系统三种模式。
- 主题切换即时生效并自动保存。
- 颜色不仅依赖色相表达状态，需同时使用文字和图标，满足可识别性。
- 深色主题不能使用纯黑大面积背景，保证层级和文字对比度。

### FR-12 系统托盘

- 关闭主窗口默认最小化到托盘，不退出应用。
- 托盘图标右键菜单：打开主界面、开始/恢复在线、开始挂起、显示今日摘要、退出应用。
- 托盘 tooltip 显示当前状态和今日在线时长。
- 退出应用需要保存成功后退出；保存失败时提示用户选择重试或退出。
- 托盘提示可在设置中关闭。

### FR-13 班次自定义

- 支持新增班次名称、开始时间、结束时间、目标在线时长、是否跨午夜。
- 支持为每个星期绑定班次。
- 校验结束时间必须晚于开始时间，除非启用跨午夜；班次最长不超过 24 小时。
- 目标在线时长必须大于 0 且不超过班次总长；允许用户开启“班次外补时”。
- 删除班次前提示该班次历史记录不受影响。
- 修改班次后，对历史记录采用历史快照，对未来日期采用新配置。

## 9. 交互与视觉要求

### 9.1 Windows 11 风格

- 使用清晰的层级、轻边框、适度圆角和一致的间距体系。
- 优先使用 Fluent 风格控件和系统行为：标题栏、按钮、开关、下拉框、日期选择器、通知。
- 主操作按钮应明显但不占据过多空间；挂起与恢复必须有清晰的状态差异。
- 重要数字优先于装饰元素，主界面避免堆叠大量卡片。
- 所有图标按钮提供可访问名称和悬停提示。

### 9.2 状态颜色与文案

## 11. 异常、边界和恢复规则

1. **应用崩溃**：启动时读取最后保存状态和最后心跳时间，恢复前弹窗显示将补记的时间区间，由用户确认。
2. **电脑睡眠/锁屏**：不根据系统状态自动补记，用户恢复后主动选择继续在线或挂起。
3. **系统时间被修改**：发现时间倒退或跳跃超过 5 分钟时暂停自动补齐，并提示校准。
4. **重复点击**：按钮操作必须幂等，连续点击不能生成重叠区间。
5. **班次外启动**：显示当前不在班次内；如果当日目标未完成，默认允许继续在线补足目标。
6. **超过 24 小时未结束**：启动跨天结算；普通班次只结算到班次边界，跨午夜班次按配置结算。
7. **数据损坏**：启动时保留最近 3 份本地备份，尝试从最近有效备份恢复，并提示用户。
8. **多个应用实例**：启动互斥锁，第二次启动只激活已有窗口，不创建第二个计时器。
9. **时区**：V1.0 使用系统本地时区；导出时间也使用本地时区。
10. **时长精度**：底层按秒保存，展示按秒；统计和 Excel 可按分钟展示，但不得丢失原始秒数。

## 12. 通知与提醒

默认关闭打扰性通知，仅在以下场景提醒：

- 班次开始前 5 分钟：提示今日班次。
- 挂起余量剩余 15 分钟：提示挂起余量不足。
- 挂起参考已超出：提示已超出，并显示超出时长。
- 今日在线已达标：提示已达标。
- 班次结束仍未达标：提示仍需补时。
- 数据保存失败：提示检查存储目录。

所有提醒均可单独关闭，提示不得抢占全屏或强制置顶主窗口。

## 13. 技术与非功能要求

### 13.1 技术建议

- 桌面容器：Tauri 2 或同等轻量 Windows 桌面方案。
- 前端：React + TypeScript，组件采用 Fluent 风格设计系统。
- 本地存储：SQLite，配合事务写入和迁移机制。
- Excel：使用支持 `.xlsx` 的本地库，导出不依赖网络。
- 系统能力：Windows 托盘、通知、开机启动、窗口置顶、单实例锁。

技术方案不是产品强制项，但必须满足本地优先、低资源占用和离线可用。

### 13.2 性能

- 空闲状态 CPU 占用目标小于 1%，内存占用目标小于 150 MB。
- 主界面首次打开目标小于 2 秒。
- 计时更新不能造成界面明显卡顿。
- 应用不依赖网络完成核心计时和统计。

### 13.3 安全与隐私

- V1.0 数据默认只存储在本机，不上传第三方服务。
- 不采集客服聊天内容、客户信息、屏幕内容或键盘输入。
- 导出文件由用户主动选择路径生成。
- 设置和数据备份文件应避免明文暴露不必要的系统信息。

## 19. 产品结论

本产品的核心不是限制用户休息，而是把“班次可用时间”“有效在线时间”和“挂起参考时间”分开记录，让用户随时知道当前完成情况。V1.0 应优先保证计时口径清楚、数据可恢复、日结和统计可信，再扩展团队协作与第三方平台对接。
